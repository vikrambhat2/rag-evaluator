# Chunking Strategies for RAG Systems

## Why Chunking Matters

Chunking is the process of breaking down large documents into smaller, semantically meaningful pieces before embedding and storing them. The quality of chunking directly affects retrieval quality. Chunks that are too large may contain irrelevant information that dilutes the signal. Chunks that are too small may lose important context needed to answer questions.

## Fixed-Size Chunking

The simplest approach splits text into chunks of a fixed number of tokens or characters. A typical configuration uses 500 tokens per chunk with a 50-token overlap between consecutive chunks. The overlap ensures that information at chunk boundaries is not lost.

**Advantages**: Simple to implement, predictable chunk sizes, easy to reason about storage costs.

**Disadvantages**: May split sentences or paragraphs in the middle, breaking semantic coherence. A sentence about "the results showed improvement" might end up in a different chunk from the data it references.

## Recursive Character Splitting

This strategy attempts to split text along natural boundaries. It tries splitting by paragraphs first, then by sentences, then by words, and finally by characters. At each level, it checks whether the resulting chunks are within the size limit. This produces chunks that better preserve semantic boundaries.

LangChain implements this as `RecursiveCharacterTextSplitter`, which accepts a list of separators (e.g., `["\n\n", "\n", ". ", " "]`) and recursively applies them. This is one of the most commonly used chunking strategies in production RAG systems.

## Semantic Chunking

Semantic chunking uses embeddings to determine where to split text. It computes embeddings for each sentence, then measures the similarity between consecutive sentences. When the similarity drops below a threshold, it creates a chunk boundary. This approach ensures that each chunk contains semantically coherent content.

The main disadvantage is computational cost. You need to embed every sentence before chunking, which can be slow for large document collections. However, it produces higher-quality chunks that often lead to better retrieval performance.

## Document-Structure-Aware Chunking

This strategy uses the document's own structure (headings, sections, tables, lists) to determine chunk boundaries. For markdown documents, it splits at heading boundaries. For HTML, it uses DOM structure. For PDFs, it uses layout analysis to identify sections.

This approach works especially well for well-structured documents like technical documentation, research papers, and knowledge base articles. It preserves the author's intended information groupings.

## Chunk Size Selection

Choosing the right chunk size depends on several factors:

- **Embedding model context window**: Most embedding models have a maximum input length (e.g., 512 tokens for many BERT-based models, 8192 for nomic-embed-text). Chunks should not exceed this limit.
- **Query type**: Short factual queries work well with smaller chunks (200-500 tokens). Complex analytical queries may benefit from larger chunks (500-1000 tokens) that provide more context.
- **Retrieval top-k**: If you retrieve more chunks (higher k), smaller chunks work better because you aggregate more diverse information. With lower k, larger chunks ensure each retrieved piece carries enough context.

## Overlap Configuration

Chunk overlap is the number of tokens shared between consecutive chunks. An overlap of 50-100 tokens is common. Overlap helps in two ways:

1. It prevents information loss at chunk boundaries.
2. It provides retrieval redundancy, meaning a relevant passage near a boundary has two chances to be retrieved.

However, excessive overlap increases storage costs and can lead to duplicate information in retrieved context. A good rule of thumb is 10-20% of the chunk size.

## Metadata Enrichment

Regardless of the chunking strategy, enriching chunks with metadata improves retrieval. Common metadata includes:

- **Source document**: Title, author, date, URL
- **Position**: Section heading, page number, chunk index
- **Relationships**: Links to parent, sibling, or child chunks
- **Summary**: A brief description of the chunk's content generated by an LLM

Metadata enables hybrid retrieval strategies that combine semantic search with metadata filtering, such as "find chunks about embeddings from documents published after 2023."
